# /task — タスク着手フロー

タスクに着手する際の標準フロー。AskUserQuestion によるヒアリング → 仕様書作成 → レビュー → 実装 → docs更新 の順で進める。

## 使い方

```
/task <タスク名または説明>
```

引数を省略した場合は `docs/tasks.md` の高優先度未着手タスクを候補として提示する。

---

## フロー

### Step 1: ヒアリング（AskUserQuestion × 3〜4問）

コードと docs からタスクの背景・現状を読み取り、判断できない重要な分岐点のみを AskUserQuestion で確認する。
確認すべき典型的な観点:
- **スコープ**: 何を変更・追加・削除するか（例: どのモードに適用するか）
- **UI/UX仕様**: 見た目・挙動の詳細（例: 表示条件・アニメーション有無）
- **非機能要件**: エラー処理・パフォーマンス・後方互換性
- **優先度・制約**: 他機能との依存・段階リリースの有無

コードを読めば分かることは聞かない。

### Step 2: 仕様書作成

`docs/specs/{feature_name}_spec.md` に仕様書を作成する。

```markdown
# {機能名} 仕様書

> 作成日: {date} / ステータス: Draft

## 概要
（1〜2行で目的・背景）

## 要件
### 機能要件
- ...

### 非機能要件
- ...

## UI/UX仕様
（画面・コンポーネント・状態ごとの挙動）

## 実装方針
（変更するファイル・主な変更内容）

## 完了条件
- [ ] ...
```

### Step 3: 仕様書レビュー（AskUserQuestion × 1問）

仕様書の内容をユーザーに提示し、「この仕様で実装してよいか」を確認する。
修正があれば仕様書を更新してから実装に進む。

### Step 4: 実装

仕様書の完了条件に沿って実装する。

### Step 5: docs更新

- `docs/tasks.md`: 該当タスクを `[x]` に変更
- `docs/product.md` / `docs/development.md`: 機能変更があれば更新

---

## 注意事項

- ヒアリングで得た情報はすべて仕様書に反映してから実装する（チャット内のみで判断しない）
- 仕様書レビュー前に実装を開始しない
- 実装中に仕様と異なる判断が必要になった場合は都度ユーザーに確認する
