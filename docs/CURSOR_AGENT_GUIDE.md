# Cursor Agent 活用ガイド — トークン節約 & 開発効率化

## 目的

Cursor Agent のリクエスト消費を抑えながら、開発スピードと品質を両立するための開発者向けリファレンス。

---

## 1. トークン消費の仕組み

Agent は会話ごとに記憶をリセットする。毎回「プロジェクトの前提知識 + 会話履歴 + 依頼内容」をコンテキストとして組み立てるため、**無駄な情報が増えるほど消費量が増える**。

### ツール別消費量の目安

| ツール | 消費トークン目安 | 適切なユースケース |
|---|---|---|
| Tab 補完 | 500〜2,000 | 単語・行の自動補完 |
| ⌘K Inline Edit | 1,000〜3,000 | 選択コードの局所修正 |
| Chat（Ask Mode） | 2,000〜5,000 | 質問・調査・コード説明 |
| Agent（実装） | 4,000〜8,000 | 複数ファイルの実装・修正 |
| **やり直し1回** | **+4,000〜8,000** | 要件食い違い・規約違反 |
| **不要ファイル読込** | **+数千〜数万** | コンテキスト汚染 |

**最大の節約** = やり直しをゼロにすること。1回のやり直しで Agent セッション1回分を無駄にする。

---

## 2. ツール使い分けガイド

```
依頼の複雑さ
    │
    ├─ 1〜数行の修正         → ⌘K (Inline Edit)
    │
    ├─ 質問・コード調査      → Ask Mode (読み取り専用)
    │
    ├─ 1ファイル内の改修     → ⌘K または Agent
    │
    ├─ 複数ファイルの実装    → Agent + Shift+Tab (Plan Mode)
    │
    └─ 繰り返しワークフロー  → /コマンド (/pr /refactor /new-feature)
```

### Plan Mode（Shift+Tab）を使うべきタイミング

- 新機能追加（複数ファイルを触る）
- リファクタリング（影響範囲が広い）
- 初めて触る機能（Agent が計画を立ててから実装）

Plan Mode では Agent が調査 → 計画提示 → 承認後に実装するため、**方向違いの実装によるやり直しを防げる**。

## 3. 効果的なプロンプトの書き方

### 基本パターン

```
[動詞] + [対象] + [制約]
```

| 悪い例 | 良い例 |
|---|---|
| `filter を改善して` | `filter feature のマッチ数表示が0のとき apply ボタンを無効化して。UI の見た目は変えない。` |
| `コードを整理して` | `/refactor lib/apps/it_pass/features/filter/` |
| `バグを直して` | `FilterViewModel.apply() で matchCount が null のまま遷移するバグを直して。テストは追加しない。` |

### ファイルの @タグは最小限に

Agent は grep・セマンティック検索でファイルを自動検索する。**タグしすぎると無関係ファイルがコンテキストに入り精度が下がる**。

```
❌ @filter_sheet.dart @filter_view_model.dart @quiz_repository.dart を参考に...
✅ filter feature の○○を修正して（Agent が自動で関連ファイルを探す）
```

### 会話の分け方（新チャットの判断）

**方針:** 依頼の**主題が変わった**ときは **新しいチャット** がおすすめ。前の話題が混ざると実装がブレ、会話が長いほど**入力トークン**も増える。

| 状況 | 推奨 |
|---|---|
| 別件（例: filter 実装 → PR 作成 → 別 feature） | **新チャット** |
| 1 つの論理単位が完了（例: 1 PR 分の実装が終わった） | **新チャット** |
| 往復が多い（目安 **15〜25** 超、または指示と実装の矛盾が増えた） | **新チャット** |
| Agent が同じミスを繰り返す | **新チャット** |
| 同じ件の続き（同一バグの追跡・同一ファイルの微調整） | **同一チャットで可** |

#### 新チャットに移るときの文脈の引き継ぎ

1. **`@Past Chats`** で必要な過去スレだけ参照する（長文コピペより節約になりやすい）
2. **ブランチ名 + やったこと一言**（コミット済みなら）を最初のメッセージに書く
3. **制約・完了条件**を 3〜5 行のメモにして貼る
4. Plan Mode で保存した計画がある場合は **`.cursor/plans/` のファイルパス**を伝える

## 4. 実装前セルフチェックリスト

依頼を送る前に以下を確認する。Agent にも自動でヒアリングさせる（`pre-implementation.mdc` ルール）が、自分でも事前に整理しておくと会話が短くなる。

```markdown
- [ ] 変更範囲は1 feature 以内に絞れているか
- [ ] 完了条件は明確か（例: flutter analyze 0件 + ○○の動作確認）
- [ ] 禁止事項を伝えたか（テスト追加しない / パッケージ追加しない / UI変更しない 等）
- [ ] 変更単位を決めたか（原則1ファイルずつ）
- [ ] 中間確認ポイントを決めたか（例: 計画提示後・各ファイル変更後）
- [ ] 単純な修正なら ⌘K で済まないか
- [ ] Plan Mode（Shift+Tab）で計画を先に立てたか（複数ファイルの場合）
- [ ] 依頼が別件なら **新チャット** に切り替えたか（§3 参照）
```

## 5. 過去の会話を再利用する

（会話をいつ切るかは **§3 会話の分け方** を参照。）

新しい会話で「前回の続きで…」と長く説明するのはトークンがかさむ。代わりに:

```
@Past Chats で「filter機能」の会話を参照して
```

Agent が必要な部分だけを選択的に読み込む。

## 6. 現在の `.cursor/` セットアップ一覧

### Rules（自動適用）

| ファイル | 適用タイミング | 目的 |
|---|---|---|
| `agent-operating.mdc` | **常時** | トークン節約の判断基準・Agent 応答を簡潔に |
| `pre-implementation.mdc` | **常時** | 実装前ヒアリング強制 |
| `project-overview.mdc` | **常時** | プロジェクト概要・コマンド |
| `dart-flutter-conventions.mdc` | Dart ファイルを開いたとき | Widget/Riverpod/コメント規約 |
| `error-handling.mdc` | ViewModel/Repository 編集時 | エラー処理パターン |
| `it-pass-domain.mdc` | `lib/apps/it_pass/**` 編集時 | ドメイン知識 |

### Commands（`/` で呼び出し）

| コマンド | 用途 |
|---|---|
| `/new-feature it_pass xxx` | 新 feature のスケルトン生成 |
| `/refactor lib/apps/...` | リファクタリング（7フェーズ手順） |
| `/fix {ログ}` | バグ修正（仮説 → 最小変更） |
| `/pr` | コミット + PR 作成を自動化 |

### Skills（関連依頼で自動発動）

| Skill | トリガー |
|---|---|
| `flutter-refactor` | 「リファクタして」「整理して」と言ったとき |
| `debug-flutter` | 「バグ」「エラー」「クラッシュ」「analyze」等 |

### MCP（Agent が自動で利用）

| サーバー | 用途 |
|---|---|
| `dart` | pub.dev 検索・Dart API ドキュメント参照 |
| `firebase` | Firestore / Auth / Hosting 操作 |
| `figma-developer-mcp` | デザインファイルからコード生成 |
| `github` | PR / Issue / コード検索（要 PAT 設定） |

## 7. トークン節約の優先順位まとめ

| 優先度 | 施策 | 節約効果 |
|---|---|---|
| ★★★ | 実装前にヒアリングして要件を固める | やり直し1回 = 8,000トークン節約 |
| ★★★ | Plan Mode で計画を承認してから実装 | 方向違い実装を防ぐ |
| ★★★ | 単純修正は ⌘K を使う | Agent の1/3〜1/5 のコスト |
| ★★ | 不要なファイルを @タグしない | コンテキスト汚染を防ぐ |
| ★★ | 会話を適切に区切る | 精度低下・ノイズ蓄積を防ぐ |
| ★★ | `/コマンド` で繰り返し作業を自動化 | 毎回の説明ゼロ |
| ★ | `@Past Chats` で過去の文脈を再利用 | コピペ説明ゼロ |

## 8. Cursor User Rules に貼る例（全プロジェクト共通）

Cursor Settings → Rules → **User rules** に、次のいずれか（または両方）を貼ると、チャット全体で応答が短くなりやすい。

```text
応答は日本語で簡潔に。前置き・締めの挨拶・称賛は不要。見出しと箇条書き中心。
コード変更を伴う依頼では、説明は1〜2文にし、主に変更差分・ファイルパス・必要なコマンドのみ示す。長い図表や一般論は避ける。
```

Agent の「判断」はリポジトリの `.cursor/rules/agent-operating.mdc` が優先される。User Rules は会話の口調・長さの補助として使う。
