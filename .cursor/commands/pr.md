# /pr — PR 作成

現在の変更をコミットして GitHub に PR を作成します。

## 手順

### Step 1: 変更内容を確認する
```bash
git status
git diff HEAD
```
- 変更ファイルと差分を確認する
- 未追跡ファイル（??）が意図的なものかを確認する

### Step 2: ステージングする
```bash
git add .
```
`.gitignore` に含まれるべきファイル（`.env`、シークレット、生成物）が含まれていないか確認する。

### Step 3: コミットメッセージを生成する

変更内容を分析して日本語でコミットメッセージを作成する。

**フォーマット:**
```
{変更の種類}: {何をしたかを簡潔に}

{任意: 変更の背景・理由（コードを見ただけでは伝わらない場合のみ）}
```

**変更の種類:**
- `feat:` 新機能追加
- `fix:` バグ修正
- `refactor:` リファクタリング（機能変更なし）
- `style:` コードスタイル修正（動作変更なし）
- `chore:` ビルド設定・依存関係の更新
- `docs:` ドキュメント変更

**例:**
```
feat: フィルター機能に習熟度レベル絞り込みを追加
fix: 解答後に次の問題へ進めなくなるバグを修正
refactor: FilterViewModel の matchCount 計算を非同期処理に移行
```

```bash
git commit -m "{生成したメッセージ}"
```

### Step 4: リモートにプッシュする
```bash
git push origin HEAD
```
初回プッシュの場合:
```bash
git push -u origin HEAD
```

### Step 5: PR を作成する
```bash
gh pr create \
  --base main \
  --title "{コミットメッセージと同じタイトル}" \
  --body "$(cat <<'EOF'
## 変更内容
{変更ファイルと差分から箇条書きで変更内容を説明}

## 確認方法
{レビュアーが動作確認する手順（任意）}
EOF
)"
```

### Step 6: PR URL を返す
作成した PR の URL をユーザーに表示して完了。

## 注意事項
- `flutter analyze` のエラーが残っている場合は PR 作成前に警告する
- 秘密鍵・APIキーが含まれていないか git diff で必ず確認する
- ブランチ名が `dev` の場合はそのまま PR を作成する（feature ブランチからの PR を推奨）
- `main` への直接 push は行わない
